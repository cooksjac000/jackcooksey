G.R.I.E.V.O.U.S.
Robotics Research Lab

In the Fall 2025 semester, I have been working as a research assistant in the Autonomous Systems Group. There, I'm working with my PI, Alexander Nettekoven, to develop a low-cost household robot. This robot, G.R.I.E.V.O.U.S., is designed to assist with household chores and other menial tasks. So far, I've built and programmed Grievous to work in teleop, and now we're moving on to training an AI model with our data.

Problem

Develop a low-cost robot with humanlike functionality that can operate in modern living spaces and perform household duties. The robot must be able to reach table/kitchen counter height in the Z-axis and reach 0.35m in the X and Y axes. The ultimate goal is to develop a robot that can run autonomously, without human supervision.

Design Process

At the start of this project, we researched different humanoid and human-like robots to determine which design would best serve our use case. We examined robots such as the HopeJR, MobileAloha, XLeRobot, and SO101, which the XLeRobot utilises. From our research, we determined that the XLeRobot would provide a good baseline for us to improve on. So we used it and later added two SO101 Leader arms to its back, like the MobileAloha, for teleop, and much later, we decided to add two SO101 Follower arms to the bottom to access the ground without needing a Z-axis movement solution.

Design Process

With this idea in mind, I assembled some of the SO101 arms and the XLeRobot head mechanism, but my main focus was on the software setup. I worked with a Raspberry Pi 5 for the first time and successfully set up a headless environment that we could remotely SSH into from our machines. I then set up the Lerobot environment with the XLeRobot robot and had to fix a few errors before we got a successful keyboard teleop running. With this baseline, we decided to use as much of the Lerobot architecture as we could, so I made a custom script that mimicked the teleoperate script provided by Lerobot, but separated the observation function to improve control frequency, now getting a constant 60 Hz even with 3 cameras running at 30 fps, and sent the observation over ZMQ to a remote computer.

Design Process

At this point, we realised that the parts of the robot assembled by a different research assistant were not up to scratch, so I got to work fixing them and making sure Grievous could operate for an extended period of time. I made sure to tighten every screw and use at least 3 screws for parts that wouldn't need to be taken off again. I also used Loctite to attach the grippers, which were especially prone to getting loose.

Design Process

With the teleop functional, the next goal was to record the robot's state during teleop so that we could train a VLA model on our own data. We decided to use the provided Lerobot recording script. To do this, we needed to make a robot and a teleoperator to work with the Lerobot script. My PI designed the robot's basic structure, while I made it work and built the teleoperator. The robot contained a host and a client script, the host being run on the Raspberry Pi, and the client on a remote machine. The host runs the arm teleop control while also pulling observations from the cameras and arms to send over ZMQ to the client. The client then takes the observations and processes them so the Lerobot recording script can record the data and send it to the Hugging Face repository for storage. The teleoperator also runs on the remote machine, mimicking a real teleoperator, but its sole purpose is to pull action data from the ZMQ socket and send that to the recording script, so we don't have to edit what is provided by Lerobot.

Major Challenge: Unreliable Motors

The motors we are using are Fingertech STS3215 high-torque servos. From using them, we've discovered that they are liable to fail under some circumstances. For a while, I was running our teleop at less than 30 Hz, which caused a lot of jittering in the motors. This led to them repeatedly failing, causing our record to crash and taking precious time to replace. To fix this, I was able to increase the teleop frequency to 60 hz, and now the motors run just fine. But this could be a sign that they won't work for long-term use without human supervision, so we are looking into ways to mitigate damage to the motors.

Results

So far, I've successfully created a recording pipeline where a user is able to teleop Grievous from an onboard Raspberry Pi host, with 60 Hz control and observation frequency, sending information to a remote client for processing and storage. I will continue my involvement in the Autonomous Systems Group next year, and continue to produce results.

Future Considerations

The next stage of this project is to train a VLA model and deploy Grievous in a household setting.
